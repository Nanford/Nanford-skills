# 输入归一指南

把口语化需求、会议纪要、聊天记录、需求文档整理成统一的需求事实层——不是总结文章，而是提炼「可驱动页面设计」的内容。

## 输入类型识别

### A. 口语化需求
特征：描述简短、跳跃、不完整。
提取策略：大胆补全角色和核心任务，积极使用默认推断。

### B. 会议纪要
特征：多人观点、待办项、争议点、优先级。
提取策略：抓结论、标争议、忽略寒暄和组织安排。

### C. 半结构化需求
特征：已有模块/角色但缺页面和交互。
提取策略：补全页面清单和状态流转。

### D. 目标导向型需求
特征：只有业务目标，无产品形态。
提取策略：从业务目标反推产品形态和角色。

### E. 需求文档
特征：PDF/Word/Notion 等结构化文档。
提取策略：提取功能列表和用户故事，忽略项目管理信息。

## 输入质量评估

| 等级 | 条件 | 推断策略 |
|------|------|----------|
| 高 | 角色、目标、核心动作均明确 | 保守推断，严格跟随用户表述 |
| 中 | 有目标和部分模块，缺交互 | 适度推断页面结构和交互 |
| 低 | 只有模糊目标或一句话 | 积极推断，最小可行方案先行 |

## 提取优先级

按顺序提取：
1. 谁在用（角色）
2. 要完成什么任务（核心动作）
3. 页面上必须出现什么（数据对象）
4. 关键动作是什么（操作）
5. 结果如何反馈（状态反馈）
6. 权限、状态、异常处理

## 会议纪要处理规则

**重点抓：**
- 已达成一致的产品结论
- 被多次提到的高优先级事项
- 影响页面范围和交互的争议点
- 明确提到的角色、模块、流程、字段

**忽略：**
- 寒暄、组织安排
- 与页面设计无关的汇报性内容
- 不影响原型的背景细节

**多人意见冲突：**
1. 有结论的采用结论
2. 无结论的采用保守方案
3. 争议标记为「待确认项」

## 统一输出模型

所有输入统一归为三类：

**已确认事实** — 用户明确说过或会议有结论。

**默认推断** — 用户没说清但根据产品常识可补全。标记格式：`[默认推断] 理由`。

**待确认项** — 会影响页面结构或流程的关键不确定项。最多保留 5 条高影响项。

## 口语映射表

| 口语表达 | 解读为 |
|----------|--------|
| "先看个大概效果" | 首轮优先出首页/总览页 |
| "能不能也看到异常和处理" | 需要异常列表 + 详情层 + 处理动作 |
| "老板主要看整体" | 必须有总览页，信息聚合优先 |
| "操作的人也要能处理" | 需要列表页和详情/处理层 |
| "这个先简单点" | 减少页面数，不减少核心链路 |
| "类似 XXX 那样的" | 参考竞品布局，但不照搬 |
| "要能导出" | 列表/报告增加导出按钮 |
| "要有权限" | 增加角色区分和权限边界 |
| "移动端也要能用" | 优先响应式或独立移动端视图 |
| "要有审批" | 增加审批流程和状态流转 |
| "先不做那么复杂" | 首轮只做主线，不做分支 |
| "给客户看的" | 偏演示型，视觉优先于功能深度 |
| "内部用的" | 偏效率型，功能优先于视觉 |
| "有没有通知" | 需要消息/通知模块 |
| "要有统计" | 增加数据统计卡片或图表 |
| "像那个管理后台" | B 端后台模式，左导航 + 表格 |
| "先把最核心的做出来" | 只做 P0 页面 |
| "加个搜索" | 列表页增加搜索栏 |

## 冲突处理优先级

1. 业务目标 > 视觉偏好
2. 核心角色高频任务 > 低频管理动作
3. 首轮原型主链路 > 边角功能
4. 不确定时走保守方案并标记
