# Nanford-skills

一个持续沉淀个人高频工作流与实战方法论的 Skills 仓库。

这个仓库不是单纯存放提示词，也不是零散的灵感备忘录，而是把日常可复用、可交付、可迭代的能力，逐步整理成结构化 Skill，方便在内容创作、方案策划、AI 办公、项目沟通和知识沉淀中重复调用。

当前仓库的定位很明确：先从我自己高频使用、已经验证有效的技能开始整理，后续再根据实际工作与内容创作需求持续扩展，形成一套真正服务个人生产力的 Skill 资产库。

---

## 这个仓库会收录什么

Nanford-skills 主要收录三类内容。

第一类是内容创作类 Skill，重点解决公众号文章、爆款内容拆解、选题分析、表达结构、风格复刻、信息差内容提炼等问题。

第二类是 AI 办公与效率类 Skill，重点沉淀那些能直接提升输出效率的流程能力，比如文档整理、方案生成、培训内容拆解、提示词结构化、会议材料整理、知识库沉淀等。

第三类是业务与项目类 Skill，面向我日常的实际工作场景，包括售前方案、项目调研、需求拆解、行业材料分析、汇报逻辑梳理、行业化表达模板等。

随着仓库持续更新，后面还会补充更多贴近真实业务和个人创作场景的实用 Skill。

## 当前已收录 Skill

### 1. viral-article-analyzer

这是当前仓库上传的第一个 Skill，定位是“公众号爆款文章深度拆解系统”。它主要服务于公众号文章分析、爆款逻辑拆解和写作方法复盘。

这个 Skill 的核心价值，不是简单总结一篇文章写了什么，而是进一步回答：为什么它有效、它打动了谁、它用的是什么内容结构、说服路径、情绪钩子和传播机制，以及这些能力如何迁移到下一篇文章里继续使用。

它适合以下场景：

当我拿到一篇公众号文章，想判断它为什么能爆；当我想系统学习一篇文章的标题设计、开头留存、情绪价值、金句结构和互动钩子；当我希望把“看过一篇好文章”的感受，沉淀成可复用的写作套路时，这个 Skill 就可以直接派上用场。

从目前的设计来看，`viral-article-analyzer` 支持以下输入方式：正文粘贴、文章链接、截图内容，或者只有标题的简化输入。它的输出会围绕标题解剖、开头留存、核心观点、说服策略、情绪触发点、金句提炼、语言风格、互动钩子等维度展开，目标是把一篇文章拆成创作者能直接复用的方法论。

## 仓库结构建议

为了方便后续持续扩展，建议仓库保持下面这种相对清晰的结构：

```text
Nanford-skills/
├── README.md
├── skills/
│   ├── viral-article-analyzer.md
│   ├── skill-name-2.md
│   └── skill-name-3.md
├── assets/
│   └── screenshots/
└── docs/
    ├── changelog.md
    └── usage-notes.md
```

其中，`skills/` 用来存放正式 Skill 文件；`assets/` 用来存放截图、示意图或案例素材；`docs/` 用来记录更新日志、使用说明、参数配置方式和后续规划。

如果前期你想先轻量化管理，也可以先只保留 `README.md + skills/` 两层结构，等数量多了再扩展。

## Skill 的统一写法建议

为了让后续仓库越来越整齐，每个 Skill 最好都保持相对统一的结构。一个完整 Skill 建议至少包含这几个部分：名称、用途说明、触发条件、适用场景、输入方式、输出要求，以及特殊情况处理。

如果某个 Skill 未来需要额外配置参数，比如角色风格、输出长度、行业类型、目标平台、文章语气、是否联网抓取、是否启用某种分析维度等，建议在 Skill 文件中单独增加一个“参数配置”章节。同时，README 也会同步新增一节“参数说明”，集中告诉使用者：哪些 Skill 需要额外参数、参数是什么意思、如何填写。

也就是说，当前 README 先以“仓库导航页”为主；后面随着 Skill 越来越多、越来越复杂，再逐步升级为“仓库首页 + Skill 索引 + 参数文档入口”的组合型说明页。

## 当前使用方式

如果只是个人使用，这个仓库最适合两种用法。

一种是把它当成自己的 Skill 资产库，后续新增任何常用 Skill，都统一沉淀进来，避免提示词散落在聊天记录、Notion、文档和本地文件里。

另一种是把它当成一个持续演进的个人方法论仓库。每新增一个 Skill，不只是多一个工具，更是在沉淀一个被验证过的工作流。时间拉长之后，这个仓库本身就会变成你的 AI 工作系统。

---

## 后续更新原则

这个仓库后续会按“先实用，再标准化，再体系化”的思路持续更新。

前期优先上传那些我日常高频使用、能马上解决问题的 Skill；中期再统一整理命名规范、目录结构、参数配置和版本记录；后期逐步形成一个覆盖内容创作、AI 办公、业务表达、项目分析等多个方向的个人 Skill 工程。

如果后续某些 Skill 开始依赖明确参数，我会同步更新本 README，把相关 Skill 的配置项、默认值、示例填写方式补充进来，保证整个仓库不是“只看不懂”，而是拿来就能用。
